# Architecture Notes

This file keeps user-owned architecture notes that are not yet ready to be
folded into the main handoff or implementation plan.

## Notes

1. При получении перечня смс мы также должны проверить, совпадает ли picked_sms с текущей (т.е. не было ли еще неисполненного желания пользователя сменить отображаемую смс). Если да - то по окончанию процесса вызываем стандартную процедуру обработки изменения для топика /on контрола пикера.
По тем же соображениям нам явно нужен первый "мутекс" в нашем коде: после получения перечня смс мы должны заблокировать обработку изменения для топика /on контрола пикера до завершения обработки нового перечня СМС. Однако сейчас он реализуется архитектурно: обработкой событий занимается один и тот же loop, и эти два обработчика не могут запуститься одновременно, что нас полностью устраивает.

2. Итог рассуждений по обработке обновления перечня SMS: picked_sms_index трактуем как позиционное желание пользователя, а не как привязку к конкретному DBus-id. При получении нового перечня до замены списка запоминаем old_picked_sms_id, после замены списка и нормализации picked_sms_index получаем new_picked_sms_id. Новый snapshot надо запрашивать тогда, когда old_picked_sms_id != new_picked_sms_id и new_picked_sms_id существует (это универсальное решение: если еще не успели обработать пользовательский запрос на смену позиции СМС - обновим снапшот, нет запроса на мену позиции, но для текущей позиции сменился sms_id - тоже запрашиваем снапшот).При этом смена displayed_sms_id сама по себе не является критерием запроса snapshot; она остается индикатором того, какая SMS сейчас реально отображена в интерфейсе (опубликована в топики mqtt).

3. При работе с dbus/МодемМенеджером выяснилось, что он добавляет новые СМС на первый незанятый номер. То есть если удалили СМС, а потом пришла новая - она может встать на место удаленной. Соответственно, для гарантирвоанного расположения СМС в порядке поступления нельзя полагаться на их номер, как предполагалось ранее, а все равно придется подтягивать дату получения по каждой СМС и отправлять в составе измененного списка СМС.
